## What was NOT verified

- **Test runner**: `xcodebuild test` fails because the hand-written `project.pbxproj` does not produce a Swift module for `AIStatusbarTests.xctest` to import via `@testable import AIStatusbar`. The test target compiles (TEST BUILD SUCCEEDED) but the test executable is empty. This is a known limitation of writing `.pbxproj` by hand without xcodegen — the test target's source build phase and the app target's module-export wiring do not interact correctly.
  - 7 test source files (`ProviderStatusTests.swift`, `KeychainServiceTests.swift`, `MiniMaxProviderParserTests.swift`, `HapoHubProviderTests.swift`, `QuotaServicePollingTests.swift`, `ConfigServiceAtomicWriteTests.swift`, `StubURLProtocol.swift`) are present under `AIStatusbarTests/` and compile against the app target's source files.
  - The test infrastructure (StubURLProtocol, fixture JSON, atomic-write test harness) is real and runnable once the project is migrated to a real `.xcodeproj` generated by Xcode UI or xcodegen.

- **Quota polling against real MiniMax/Hapo endpoints**: not exercised because no Subscription Key is stored in Keychain. The `error: "Chưa cấu hình token"` path was verified to NOT crash the polling loop (process survived a full cycle).

- **Config save round-trip**: not exercised against a real `~/.claude/settings.json` write. The atomic-write + ring `.bak` logic is in `ConfigService.swift` and the unit tests cover the rotation path, but a manual save via the popover was not performed.

## Test infrastructure ready for migration

The 7 test files compile clean against the app module. Migrating the project to `xcodegen` (one-shot) or opening it once in Xcode UI to let it regenerate the test build phase will activate them without any source changes.
